### Unicode Homoglyphs
The technique of using alternative Unicode characters to evade filters or deceive users is often referred to as a homoglyph attack. A homoglyph is a character that looks visually identical or very similar to another character but has a different underlying Unicode code point.

For example, the Latin lowercase "a" (U+0061) looks identical in many fonts to the Cyrillic small letter "а" (U+0430). 

Simple content filters often rely on exact string matching. If a filter is configured to block the word "example", it looks for the specific sequence of Latin characters (U+0065, U+0078, U+0061, U+006D, U+0070, U+006C, U+0065). If a user replaces the Latin "a" with the Cyrillic "а", the underlying byte sequence changes. A basic filter will fail to match it, allowing the text to pass through.

### Defensive Mechanisms in Filtering
Modern content filtering and security systems employ several techniques to mitigate homoglyph evasion:

1.  **Unicode Normalization:** This is a primary defense. Standards like NFKC (Normalization Form Compatibility Composition) convert text into a standardized format before the filter processes it. NFKC maps visually similar characters (like full-width characters, circled characters, or certain ligatures) back to their standard ASCII equivalents.
2.  **Confusable Character Mapping:** Security systems maintain extensive databases of known homoglyphs across different scripts (Latin, Cyrillic, Greek, etc.). The filter translates incoming text using these maps to check for variations of blocked terms.
3.  **Character Set Restrictions:** Systems may restrict input to specific Unicode blocks relevant to the application's expected language or context, rejecting input containing unexpected scripts.
4.  **Contextual Analysis:** Advanced filters use machine learning models trained to recognize the semantic meaning or visual representation of text, reducing reliance on exact code point matching.
